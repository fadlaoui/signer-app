Architecture Approaches

How to structure multi-document signing + admin dashboard

A

File-Based (Extend Current) — Recommended

Keep JSON file storage. Each document gets its own folder with a meta.json + signatures.json + attachments. Admin auth via a hashed password file. Simplest migration from current code.

Pros

  • Zero new dependencies
  • Familiar pattern — extends what exists
  • Easy backup/restore
  • Works great for small-to-medium scale

Cons

  • No concurrent write safety
  • Slower with many documents (file I/O per request)
  • Admin accounts stored as JSON files
B

SQLite Database

Migrate to SQLite for documents, signatures, and admin accounts. Proper relational tables. Attachments stored as files, referenced from DB.

Pros

  • Proper queries and indexing
  • Concurrent access handled
  • Better for stats/aggregation
  • Standard admin account management

Cons

  • New dependency (better-sqlite3 or similar)
  • Migration from existing JSON data needed
  • Slightly more complex deployment
C

Hybrid (JSON docs + cookie-based auth)

Keep JSON files for documents/signatures (like option A) but add proper session-based auth with HTTP-only cookies and bcrypt-hashed passwords.

Pros

  • Best of both — simple storage, proper auth
  • Minimal new dependencies (just bcrypt)
  • Easy migration path
  • Secure admin sessions

Cons

  • Still no concurrent write safety
  • Two storage patterns to maintain
